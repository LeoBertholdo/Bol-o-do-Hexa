## TL;DR

Tem **um buraco de segurança crítico** que precisa fechar antes de divulgar o link: hoje, **qualquer pessoa que entre no site pode se passar pelo "Leo B." (ou qualquer outro participante) com dois cliques**. Não é teórico — é o fluxo padrão "Vincular nome". O resto é uma mistura de um bug de sync sério no mata-mata, alguns problemas de UX em mobile, e polidas. O app em si está bem montado para algo feito sem background técnico — escaping de HTML consistente, RLS habilitado, fallback offline funcional. Mas o modelo de "qualquer um cria conta e escolhe quem é" não combina com "público no GitHub Pages".

## 🔴 Crítico — fechar antes de compartilhar o link

### 1. Qualquer pessoa pode se passar por qualquer participante

**Onde:** `supabase_setup.sql` linhas 123-126 + `bolao2026.html` linhas 1422-1438 (`claimParticipant`)

A política de RLS para criar perfil é:
```sql
create policy "participants create own profile"
on public.profiles for insert to authenticated
with check (id = auth.uid() and role = 'participant');
```

Isso só checa que (a) você está se inserindo como você mesmo e (b) o role é `participant`. **Não há nenhuma checagem de que o `participant_index` que você escolheu é seu de fato.** O único bloqueio é o `UNIQUE` no banco — quem chegar primeiro pega.

**Cenário de ataque (literalmente o fluxo da UI):**
1. Atacante abre o site
2. Clica "Criar conta", coloca qualquer e-mail e senha
3. Confirma e-mail, faz login
4. Clica "Vincular nome", escolhe "Leo B." no dropdown
5. A partir desse momento, todos os palpites dele aparecem como sendo do Leo B., e o Leo B. real não consegue mais entrar — vai dar erro de "participante já vinculado"

Pior: o atacante pode pegar `participant_name` arbitrário na chamada da API (o front sempre manda `participantName(idx)`, mas RLS não força isso). Um atacante consegue rodar o INSERT direto via SDK do Supabase com `participant_name: "qualquer coisa"`, e como o `applyRemoteData` faz `next.participants[idx] = profile.participant_name`, esse nome contamina a UI de **todos** os outros usuários.

**Como consertar (escolha um):**

- **Mais simples para um bolão de amigos:** desabilita a auto-vinculação. Depois que todos criarem conta, você (admin) roda no SQL Editor do Supabase:
  ```sql
  insert into profiles (id, participant_index, participant_name, role)
  select id, 0, 'Leo B.', 'participant' from auth.users where email = 'leo@…';
  -- repete pra cada participante
  ```
  E muda a policy de insert pra `with check (app_private.is_admin())`. Os participantes nunca precisam mexer no dropdown.

- **Caminho intermediário (lista de convidados):** cria uma tabela `invited_emails(email text, participant_index int unique)` populada pelo admin antes. Muda a policy pra exigir que o e-mail do `auth.uid()` esteja na tabela e bata com o índice:
  ```sql
  with check (
    id = auth.uid()
    and role = 'participant'
    and exists (
      select 1 from invited_emails ie
      join auth.users u on u.email = ie.email
      where u.id = auth.uid() and ie.participant_index = participant_index
    )
  )
  ```

- **Mais barato (não recomendado pra público real):** desabilita signup público nas configurações de Auth do Supabase e cria as contas você mesmo, mandando "Magic links" para cada participante.

Enquanto esse buraco existir, **não compartilha o link com ninguém de fora do círculo de confiança imediato.**

### 2. Cascata de deleção do mata-mata não chega ao Supabase

**Onde:** `bolao2026.html` linhas 2486-2533 (`confirmRes`, `clearRes`, `clearDependentKnockouts`)

Quando o admin corrige um resultado de fase de grupos que invalida confrontos já preenchidos do mata-mata, `clearDependentKnockouts(id)` apaga os resultados dependentes **só no `S.results` local**. Mas `confirmRes` chama `saveResultRemote(id)` apenas para o jogo originalmente alterado. Os resultados de mata-mata continuam no servidor, e o poll de 30s os ressuscita silenciosamente — anulando a correção do admin sem aviso.

**Repro:** salva R32_01 = 1×0 → muda um resultado do grupo que invalida o confronto → R32_01 some da tela → espera 30s → R32_01 = 1×0 está de volta.

**Fix:** coleta os IDs apagados e envia delete em batch ao Supabase:
```js
function clearDependentKnockouts(id){
  const cleared = [];
  // ... lógica existente, mas push em `cleared` antes de cada delete S.results[g.id]
  return cleared;
}
// em confirmRes / clearRes:
const cleared = clearDependentKnockouts(id);
save();
saveResultRemote(id); // ou deleteResultRemote
if(remoteClient && isAdmin() && cleared.length){
  remoteWrite(() => remoteClient.from("results").delete().in("game_id", cleared));
}
```

### 3. "Publicar dados locais" zera o dono de todos os palpites

**Onde:** `bolao2026.html` linhas 1521-1556 (`pushLocalStateToRemote`)

A função faz `user_id: null` em todos os palpites importados. Combinado com a policy de update que aceita `user_id = auth.uid() OR user_id IS NULL`, **qualquer pessoa que vincular aquele `participant_index` no futuro consegue sobrescrever esses palpites**. Em conjunto com o bug #1, isso vira a receita de "atacante toma a conta + reescreve histórico do palpite".

Além disso, o botão não tem `confirm()` — clicar sem querer sobrescreve dados do servidor com a cópia local do navegador (que pode estar desatualizada).

**Fix:**
- Adicionar `if(!confirm("Isso vai sobrescrever os dados do Supabase com a cópia deste navegador. Continuar?")) return;` no topo da função
- Preservar o `user_id` original quando possível: ler primeiro o que existe no servidor e só sobrescrever o que estiver vazio, em vez de upsert cego com null
- Ou simplesmente: esconder esse botão depois que o bolão começar (tipo, depois que existir qualquer palpite no servidor com `user_id` preenchido)

## 🟠 Importante — vale arrumar logo

### 4. `confirmRes` aceita placar > 30 e deixa o banco recusar

**Onde:** linha 2486-2502

`setResDraft` capa com `Math.min(30, n)`, mas `confirmRes` lê direto do DOM (`parseInt(document.getElementById...)`)  e só valida `< 0`. Se o usuário colar "999" no input, a chamada ao Supabase falha pelo `CHECK (s1 between 0 and 30)`, toast genérico de "Não consegui conversar com o Supabase" aparece, e o `loadRemoteData` é forçado, sobrescrevendo o estado local. Confuso pra quem usa.

`setPal` (linha 2616) tem o mesmo problema para os palpites.

**Fix:** mesma clamp em ambos os lugares:
```js
const s1 = Math.min(30, Math.max(0, parseInt(...)));
```

### 5. Input perde foco a cada tecla digitada

**Onde:** `renderPalpites`, `renderJogos`, etc.

Toda a aba é re-renderizada via `innerHTML =` depois de cada `setPal`/`setResDraft`/`confirmRes`. Quem digita rápido perde o input ativo. Em mobile isso é especialmente ruim porque o teclado virtual fecha e abre.

**Fix mínimo:** depois de salvar um palpite individual, atualizar só o `badge` da linha (`document.querySelector(`#row_${gid} .sbadge`).className = …`) em vez de redesenhar a aba inteira. Ou no mínimo, fazer o re-render via `requestAnimationFrame` para deixar o input perder e ganhar foco numa única tick.

### 6. Sem proteção contra conflitos no poll de 30 segundos

**Onde:** `loadRemoteData` (linha 1318) + `applyRemoteData` (linha 1284)

O poll faz `S = migrateState(next)` — substituição total do estado. Se o usuário tem um palpite local sendo digitado quando o poll chega, perde. Não existe nenhuma lógica de "tenho gravação pendente, pula esse poll".

**Fix mínimo:** segura uma flag `let writeInFlight = false` que vira `true` no início de cada `remoteWrite` e `false` após resolver; pula a aplicação de dados remotos se a flag estiver `true`.

### 7. Backup importado sobrescreve estado online sem aviso

**Onde:** `importBackup` (linha 2830)

Carrega JSON arbitrário em `S` e salva no localStorage. Se o usuário estava online, o próximo poll vai reverter — então não é destrutivo no servidor. Mas o admin clicando "Publicar dados locais" em seguida (item #3) é o cenário ruim. Vale um `confirm` que diga "isso só atualiza este navegador; pra mandar pro servidor use Publicar".

## 🟡 Design e UX

### 8. Hierarquia visual: ação destrutiva confunde com ação benigna

**Onde:** aba Config, "Backup de dados" (linha 2742-2753)

"Exportar JSON", "Importar JSON", "Publicar dados locais no Supabase" e "Resetar competição" estão todos lado a lado, todos com `action-btn` (verde) ou `action-btn secondary` (azul claro). Resetar competição apaga tudo e é estilizado igual a exportar backup.

**Fix:** dá ao botão de reset uma classe `action-btn danger` (vermelho `#fee2e2` com texto `var(--red)`, igual ao `.btn-x`). E muda o texto pra "Apagar todos os resultados, palpites e pódios". Considera também separar o card em "Backup" e "Zona de perigo".

### 9. Nav com 7 abas fica apertado no mobile

**Onde:** linhas 912-920 (HTML) + media queries linhas 787-798

"Início, Ranking, Grupos, Jogos, Palpites, Pódio, Config" — em telas estreitas, "Pódio" e "Config" ficam parcialmente fora da tela e dependem de scroll horizontal. Sem indicador visual de que tem mais conteúdo à direita.

**Fix:** ou consolida (junta "Grupos" e "Jogos" em uma única view, ou esconde "Config" atrás de um botão de engrenagem no header), ou adiciona uma sombra/gradient no canto direito do nav indicando "tem mais →".

### 10. Confirmações nativas vs toast: inconsistência

**Onde:** linha 2527 (`confirm("Remover este resultado?")`), 2784, 2838

O app inteiro usa o `toast` customizado e cards bonitinhos, e aí pra confirmações destrutivas dispara o `window.confirm()` cinza do sistema operacional. Quebra a estética. Mais sério: o `confirm` nativo bloqueia a thread e em alguns mobile browsers tem comportamento esquisito.

**Fix:** monta um modal de confirmação inline (mesma cara do `account-panel`, com botões "Cancelar / Confirmar"). Reaproveita pros 3 lugares.

## ✅ O que está bem feito

Vale registrar — pra um app feito sem background técnico, isso é incomum:

- **`esc()` aplicado de forma consistente** em toda interpolação de string que toca dado de usuário ou DB. XSS via render foi mitigado corretamente.
- **RLS habilitado em todas as tabelas** com policies separadas pra admin e participante. O conceito está certo — só falta apertar o INSERT de profile.
- **`app_private.is_admin()`** corretamente `security definer` com `search_path` fixo (proteção contra search-path injection que muita gente esquece).
- **Estratégia local-first com fallback** quando o Supabase está indisponível.
- **Migração de chave de storage** (`OLD_STORAGE_KEYS`) preserva dados de versões antigas — usuário não perde palpite se você mudar a versão.
- **Lógica de `score()`** está matemática e logicamente correta nos 4 casos (cravada, invertida, certo, erro), inclusive nos edge cases de empate.
- **Timezone Brasil explícito** no `BR_DATE_TIME_FORMAT` com `America/Sao_Paulo` — corretíssimo pra um bolão BR.
